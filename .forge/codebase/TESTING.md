# 테스트 (Testing)

## 현재 상태: 테스트 프레임워크 없음

이 코드베이스에는 테스트 프레임워크가 **설치되어 있지 않다**. 솔직하게 말하면 자동화된 테스트 스위트가 존재하지 않는다.

확인된 사실:

- `addons/` 디렉터리가 없다 → **GUT**(Godot Unit Test) 등 어떤 테스트 애드온도 설치되어 있지 않다.
- `test/`, `tests/`, `spec/` 등 테스트 전용 디렉터리가 없다. 저장소 어디에도 `*test*` 이름의 파일이 없다.
- `project.godot`에 `[gut]` 등 테스트 관련 설정 섹션이 없다 (Godot 4.3, Forward Plus 최소 골격).
- 테스트 러너, CI 설정, 어서션 라이브러리가 전혀 없다.

따라서 "테스트를 어떻게 실행하는가"에 대한 답은 현재로선 "실행할 테스트가 없다"이다.

## 존재하는 검증 메커니즘 (테스트는 아님)

테스트 프레임워크는 없지만, 데이터 정합성을 확인하는 두 가지 인-코드 장치가 있다.

### 1. `StageDef.validate()` — 인-코드 데이터 검증

`scripts/data/stage_def.gd:41-49`에 정의된 메서드로, 스테이지 데이터의 정합성을 점검해 오류 메시지 배열을 반환한다. duration이 0 이하인지, 스폰(spawn_pool/timeline)이 모두 비었는지, 목표(objectives)가 비었는지를 확인한다. 빈 배열이면 통과. 이것은 단위 테스트가 아니라 저작/실행 시점에 호출하도록 의도된 런타임 데이터 가드다 (주석: `## 데이터 정합성 점검. 저작/실행 시 호출.`).

### 2. `tools/seed_stage1.gd` — 시드/검증용 EditorScript

`@tool extends EditorScript`로, Godot 에디터에서 수동으로 `File > Run`해 실행한다 (`seed_stage1.gd:1-7`). 스테이지1의 적 4종과 스테이지 리소스를 코드로 빌드해 `res://data/`에 `.tres`로 저장하고, 마지막에 `st.validate()`를 호출해 결과를 `print`한다 (`seed_stage1.gd:92-93`).

```
var errs := st.validate()
print("validate(stage_hwalinseo): ", "PASS" if errs.is_empty() else errs)
```

이것은 **단위 테스트가 아니다.** 손작성 `.tres`가 깨졌을 때 데이터를 재생성하는 시드 스크립트이며, `validate()` 출력은 그 부산물에 가깝다. 자동 실행되지 않고, 어서션으로 빌드를 실패시키지도 않으며, 사람이 에디터에서 수동 실행하고 콘솔 출력을 눈으로 확인하는 방식이다.

## 계획된 테스트 전략 (PLANNED — 아직 구현 안 됨)

설계 문서 `docs/11-구현-배관-보강.md` §6.4 "테스트 전략"이 향후 테스트 방향을 명시한다. 아래는 모두 **계획**이며 현재 코드에는 없다.

| 대상 | 계획된 방법 |
|------|------|
| 성능 | 500마리 벤치 씬 — 60FPS 자동 측정 (M1 게이트) |
| 데이터 | `StageDef.validate()` + 전체 `data/**.tres` 로드 스모크 테스트 (누락/깨진 참조 검출) |
| 핵심 로직 | spatial hash 질의, 넉백 변위, 혼불 전달/회복 계산 → **GUT 단위 테스트** |
| AI 거동 | 수동 플레이테스트 (자동화 과투자 금지, YAGNI) |

문서의 방침 (`docs/11` §6.4): 테스트는 핵심 수치·검증 로직에 집중하고, AI·연출은 플레이테스트로 다루며, "프레임워크 과설치 금지". 또한 §7 체크리스트는 "테스트(벤치·validate·GUT)"를 명시적으로 **후속(미완료) 항목**으로 표시한다 (`docs/11`의 미체크 항목).

요약: GUT 단위 테스트, 성능 벤치, `.tres` 로드 스모크 테스트는 전부 계획 단계이며, 현재 저장소에는 아무것도 존재하지 않는다.
