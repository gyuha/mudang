# 05. 데이터 스키마 · 저작 도구

네 핵심 목표 — **데이터 주도 + 스테이지 저작 도구**. 그릇은 **Godot Resource(.tres)**, 도구는 **EditorPlugin 독**(D17). Godot 인스펙터가 곧 1차 저작 UI다. 별도 외부 앱은 만들지 않는다(포맷 검증 후 필요 시 JSON 임포터 추가).

---

## 1. 왜 Resource인가 (D17 근거)

- 인스펙터 편집 UI **공짜**(export 변수 → 폼).
- 타입 안전 + `@export_range`로 검증 공짜.
- `.tres`는 텍스트라 git diff·리뷰 가능.
- `load()`로 런타임 즉시 사용, 핫리로드 용이.
- 저작 주체 = 개발자/기획자(Godot 보유)이므로 인에디터로 충분.

> 비개발자/모더 저작이 필요해지면, 동일 스키마를 JSON으로 import/export하는 어댑터만 추가한다. 지금은 YAGNI.

---

## 2. Resource 계층 (전체)

```
StageDef (.tres)               ← 저작 도구가 만드는 최상위 단위 = "스테이지 하나"
├─ duration: float             # survive_time 기본
├─ map_ref: PackedScene/배경
├─ objectives: Array[ObjectiveDef]
├─ spawn_pool: Array[SpawnPoolEntry]   # 시각대별 적 가중치
├─ timeline: Array[TimelineEvent]      # 고정 이벤트
├─ budget_base / budget_growth_per_sec # budget_per_sec(t)=base+growth*t
├─ soulfire_ratio: float               # 무녀:동료 드랍 비율
├─ available_companions / 강제 편성 여부
└─ reward: StageReward                 # 해금/메타 재화

EnemyDef (.tres)               ← 적 1종 (04 참조)
CompanionDef (.tres)           ← 동료 1명 (단일 리소스, D25)
├─ role_id: StringName         # 02 참조 (tank/ranged/healer/aoe) — 거동 분기
├─ 스탯·역할 튜닝 (hp/speed/dmg/attack_range/taunt_radius/kite_min/heal_per_sec/leash …)
├─ upgrade_pool: Array[CompanionUpgrade]   # 동료 3택 풀
└─ sprite_frames

MudangUpgrade (.tres)          ← 무녀 3택 카드 1종 (01§6 카테고리)
├─ category / max_level
├─ effects per level (Array)
└─ icon, 설명

DropTable (.tres)              ← 혼불/메타 재화 드랍 확률
ObjectiveDef (.tres)           ← 목표 모듈 (04§3)
StageReward (.tres)            ← 클리어 보상/해금
```

### 2.1 StageDef (핵심 — 저작 도구의 편집 대상)
```gdscript
class_name StageDef extends Resource
@export var id: StringName
@export var display_name: String
@export var duration: float = 360.0                 # 6분
@export var background: PackedScene
@export var objectives: Array[ObjectiveDef] = []
@export var spawn_pool: Array[SpawnPoolEntry] = []
@export var timeline: Array[TimelineEvent] = []
# 예산 채널: budget_per_sec(t) = budget_base + budget_growth_per_sec * t  ([04]§2.2, [09]§5)
# (저작 단순화: Curve 대신 선형 2값. 비선형이 필요해지면 저작 도구에서 Curve 옵션 추가)
@export var budget_base: float = 2.0
@export var budget_growth_per_sec: float = 0.0333
@export var max_active_enemies: int = 500            # 동시 적 상한([06])
@export_range(0,1) var soulfire_ratio: float = 0.6   # 무녀 혼불 비중
@export var unlock_requires: Array[StringName] = []  # 선행 스테이지 (선형 해금 판정, [10])
@export var recommended_roles: Array[StringName] = []# 브리핑 권장 편성 힌트 [10]
@export var unlock_companion: StringName = &""       # 첫 클리어 시 해금 동료 [10]
@export var reward: StageReward
# --- 후속 확장(슬라이스 이후, 스테이지 풀 차별화 D22 / [10]§6) ---
@export var global_rule: StageModifier               # 시야 반경·난이도 배율 등 전역 룰
@export var hazards: Array[HazardDef] = []           # 정적 해저드존(화재/독/봉인진) — 사각형+초당피해+타입(데이터). 정적형 구현됨(2026-06-25), 확산형은 후속
```

> **신규 Resource([10]§6 — 차별화 2계층 D22):** `HazardDef`는 **정적 해저드존**(zone_rect + dps + type) — *데이터*다. **정적형은 구현됨(2026-06-25, RunScene가 등록·dps 적용; 스테이지2 화재·4 독장판 등 사용 중)**; 확산(번지는) 해저드 시뮬은 후속(계층②). `StageModifier`(`vision_radius`·`enemy_dmg_mult` 등)는 전용 시스템 `시야 제한`이 읽는다. `ObjectiveDef`의 `purify_zone` `order`(순차 점거)는 전용 시스템 `점거 게이지`가 처리. 즉 **데이터 스키마는 예약하되, 계층② 시스템 3종(시야·점거·다페이즈)이 그 데이터를 해석하는 코드**다. 슬라이스(스테이지1)는 전부 없이 동작.

### 2.2 SpawnPoolEntry / TimelineEvent
```gdscript
class_name SpawnPoolEntry extends Resource
@export var enemy: EnemyDef
@export var weight: float = 1.0     # 추첨 가중치
@export var start_time: float = 0.0 # 활성 시작(초)
@export var end_time: float = -1.0  # 활성 종료(초), -1 = 끝까지
# (저작 단순화: 시간대별 변화는 항목을 여러 개 두어 표현. Curve보다 텍스트/도구 친화적)

class_name TimelineEvent extends Resource
@export var time: float                       # 초
@export_enum("SPAWN_GROUP","MINIBOSS","RUSH","OBJECTIVE","MODIFIER") var kind: int
@export var enemy: EnemyDef                    # SPAWN_GROUP/MINIBOSS
@export var count: int = 1
@export var payload: Dictionary = {}           # 유연 파라미터
```

## 3. 런타임 로더

```gdscript
# stage_loader.gd
static func load_stage(id: StringName) -> StageDef:
    return load("res://data/stages/%s.tres" % id)
```

규약: 데이터는 `res://data/{stages,enemies,companions,upgrades,droptables}/` 하위. id == 파일명. 단순 컨벤션, 레지스트리 코드 불필요(YAGNI). 전체 목록이 필요하면 디렉토리 스캔.

## 4. 저작 도구 (EditorPlugin 독)

**1단계(슬라이스 검증 전): 도구 없음.** 순수 인스펙터로 `StageDef.tres`를 손으로 만든다. 이걸로 1장이 재미있는지 먼저 증명한다(D4 원칙).

**2단계(검증 후): EditorPlugin 독 추가.** 인스펙터만으로 불편한 부분만 보강한다. 과설계 금지.

도구가 실제로 더해야 할 가치(인스펙터가 못 주는 것):
| 기능 | 이유 |
|------|------|
| **타임라인 시각 편집기** | 시각축에 이벤트를 점으로 찍고 드래그. 배열 인스펙터로는 흐름이 안 보임 |
| **예산 곡선 + 적 가중치 미리보기** | "이 시각엔 대략 몇 마리/어떤 적"을 그래프로 |
| **원클릭 플레이테스트** | 편집 중인 StageDef로 즉시 런 실행 |
| **밸런스 요약** | 총 예상 적 수, DPS 추정, 드랍 총량 |

```
EditorPlugin: add_control_to_dock(DOCK_SLOT_RIGHT_UL, StageEditorDock)
StageEditorDock:
  - StageDef 선택/생성
  - 타임라인 트랙(가로축=시간, 이벤트 노드 드래그)
  - 예산 슬라이더(budget_base/growth) + 예상 적 수 미리보기(필요 시 비선형 Curve 옵션 추가)
  - [▶ 플레이테스트] 버튼 → 해당 StageDef로 메인 씬 실행
```

> 도구의 출력은 결국 `StageDef.tres` 한 개다. 도구가 없어도 게임은 데이터로 돈다 → 도구는 "편의"지 "의존"이 아니다. 이게 D4 순서의 핵심.

---

## 5. 검증(validation)

- `@export_range`로 1차 범위 검증(공짜).
- StageDef에 `func validate() -> Array[String]`: 빈 spawn_pool, duration<=0, objectives 없음 등 점검. 도구·실행 시 호출.

## 구현 현황 (2026-06-25)

- §2 Resource 계층 전부 정의됨(StageDef/EnemyDef/CompanionDef/MudangUpgrade/DropTable/ObjectiveDef/StageReward/HazardDef/SpawnPoolEntry/TimelineEvent). `data/`로 손작성된 스테이지·적 다수 존재.
- 신설/확장 필드(구현됨):
  - **EnemyDef**: `attack_range`, `attack_period`(원거리), `summon_id`/`summon_count`(보스 다페이즈 소환 — EnemySystem이 소비).
  - **CompanionDef**: `aoe_radius`(>0이면 광역 베기).
  - **ObjectiveDef** `purify_zone` params: `pos`(Vector2)/`radius`(점거 좌표·반경 — RunScene가 소비).
  - 신규 Resource **CompanionUpgrade**(id/display_name/role_filter/max_level/targets/deltas — 동료 성장 카드).
- `StageDef.global_rule`(StageModifier)은 **선언만(예약)** — 읽는 시스템 없음. `HazardDef` **정적 해저드존은 구현됨**(RunScene 등록·dps 적용, 스테이지2/4 등 사용); 확산형은 후속.
- §4 **저작 도구 EditorPlugin 미구현**(`addons/` 없음) — "1단계 순수 인스펙터" 계획대로의 의도된 현황. 타임라인 시각 편집기·예산 곡선 미리보기·플레이테스트 버튼은 후속.
- §5 `StageDef.validate()`는 미구현(후속).
