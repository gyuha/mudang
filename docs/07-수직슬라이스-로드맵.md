# 07. 수직 슬라이스 · 구현 로드맵

D4: **핵심 루프가 재미있는지 먼저 증명한다.** 콘텐츠·저작 도구는 그 위에 붙인다. 이 문서가 "무엇을 먼저, 어떤 순서로, 무엇을 충족하면 통과"를 정의한다.

---

## 1. 수직 슬라이스 정의 — 1장 "활인서의 밤"

| 요소 | 슬라이스 범위 |
|------|--------------|
| 스테이지 | 1개(`stage_hwalinseo`), `survive_time` 6분 + `defend_target`(병자 거점) |
| 무녀 | 레버 4종 전부(오라/넉백/혼불전달/모여라) + 부활 채널 |
| 동료 | 출전 풀 3종(화랑=탱/활잡이=딜/견습무당=힐), 슬롯 2~3 (D16·D20) |
| 적 | 3종(잡귀/처녀귀신/도깨비 엘리트) + 간이 보스 1 |
| 성장 | 무녀 일시정지 3택, 동료 비정지 보류 3택 |
| 메타 | 최소(스테이지 클리어 저장, 동료 해금 1~2) |
| 저작 도구 | **없음** — StageDef.tres 손작성으로 충분 |
| 호드 규모 | 500@PC 프로파일 통과 |

슬라이스가 증명할 질문: **"적은 많고 동료는 위험하다. 내가 포지셔닝·넉백·혼불배분·집결·부활을 잘하면 모두를 살릴 수 있다 — 그게 재미있는가?"**(DESIGN §16)

---

## 2. 구현 순서 (마일스톤)

각 마일스톤은 독립 실행/검증 가능하게 자른다.

```
M0 기반(+성능 스파이크) → M1 적 시스템(API+단순 백엔드 ~150) → M2 동료AI → M3 무녀 레버
   → M4 케어 → M5 성장 → ★재미 검증★ → M-S 스케일업(500@60fps) → M6 스폰·목표 → M7 편성·메타 → M8 폴리시
```

| M | 목표 | 검증(verify) |
|---|------|-------------|
| **M0** 프로젝트 기반 | Godot 4 프로젝트, RunScene 골격, InputAdapter(PC), 카메라, 무녀 이동. **+ 성능 스파이크**(던지는 코드: 500 더미 MultiMesh + spatial hash 이동, 60fps 확인) | 무녀가 WASD로 움직이고 카메라가 따라온다 / 스파이크가 500@60fps를 보임(접근법 검증) |
| **M1** 적 시스템(단순 백엔드) | **EnemySystem API([11]§1) 확립** + 단순 백엔드(풀링 노드/배열) ~150마리, SpatialHash, 잡귀 직진(동료/거점), contact damage | 잡귀 다수가 동료/거점으로 직진, ~150마리 안정. 렌더 백엔드는 M-S에서 교체(API 뒤) |
| **M2** 동료 AI | 공통 FSM, 탱/딜/힐 역할, separation, leash | 동료 3인이 역할대로 자율 전투(탱 전열/딜 카이팅/힐 회복) |
| **M3** 무녀 레버 | 오라 감속, 넉백(클릭), 혼불 전달, 모여라 | 4 레버가 실제로 동료 생존/적 거동에 영향 |
| **M4** 케어 | 쓰러짐 타이머+UI, 근접 채널링 부활, 상실 디버프 | 동료를 쓰러뜨리고 무녀로 부활, 방치 시 상실 |
| **M5** 성장 | 무녀 EXP→일시정지 3택, 동료 EXP→비정지 보류 3택 | 레벨업으로 레버/동료가 강해짐, 흐름 안 끊김(동료) |
| **★재미 검증★** | M2~M5를 ~150마리에서 플레이테스트 | 핵심 루프가 재미있다(§3 재미 항목). **여기서 NO면 스케일업 전에 재설계 — 매몰비용 최소** |
| **M-S** 스케일업 | EnemySystem **내부**를 SoA+MultiMesh+셰이더 애니로 교체(API 불변), spatial hash 최적화, 모바일 열화 | 500마리 60FPS(PC) / 250(모바일). **게임플레이 코드 무수정** |
| **M6** 스폰·목표 | WaveDirector(타임라인+예산), EnemyDef 3종+보스, defend_target | StageDef.tres 하나로 6분 런이 굴러가고 승/패가 난다 |
| **M7** 편성·메타 | 출전 편성 화면(슬롯2, 풀3), MetaProgress 저장/해금 | 편성 선택→런→클리어→해금 저장 사이클 |
| **M8** 폴리시·검증 | 셰이더/파티클 1차, HUD, 모바일 열화 스위치, 밸런스 1패스 | 아래 §3 수용 기준 전부 통과 |

> **시퀀스 원칙(개정):** 가장 큰 리스크는 "500마리"가 아니라 "**재미있는가**"다(§5 #1). 그래서 *재미를 먼저, 최소 인프라로* 검증한다 — M1은 **EnemySystem API 경계([11]§1)**만 고정하고 백엔드는 단순하게(~150마리). 그 API 뒤에서 M2~M5가 전체 루프를 증명한 뒤, **M-S에서 백엔드만 SoA+MultiMesh로 교체**해 500@60fps를 친다(게임플레이 코드 불변 — API 경계가 되돌리기 비용을 흡수). D19는 폐기가 아니라 **시점을 뒤로** 옮긴 것. **M0의 던지는 성능 스파이크로 접근법만 미리 확인**해 M-S가 막히지 않게 한다(렌더 방식이 근본적으로 안 되면 그때 알아야 하므로).

## 3. 슬라이스 수용 기준 (Done의 정의)

전부 충족해야 "핵심 루프 증명" 통과.

- [ ] PC에서 동시 적 500, 60FPS 유지(**M-S 스케일업** 마일스톤 — 재미 검증 M2~M5와 분리)
- [ ] 무녀 4 레버가 각각 **유의미한 선택**을 만든다(넉백 안 쓰면 동료가 죽는다 등)
- [ ] 동료를 잃을 수 있고, 케어 판단(누구를 먼저 구하나)이 실제로 발생
- [ ] 무녀 3택으로 빌드가 갈리고, 동료 혼불 배분이 전략이 됨
- [ ] 6분 런에 기승전결(초반 여유→중반 압박→막판 러시/보스)
- [ ] 새 스테이지를 **StageDef.tres 하나만 추가**해 만들 수 있다(데이터 주도 증명)
- [ ] 플레이테스터 다수가 DESIGN §16 감정선("함께 밤을 넘겼다")을 보고

## 4. 슬라이스 이후 (확장, 이 문서 범위 밖)

검증 통과 후 순서: 저작 도구(EditorPlugin) → 탈쓴퇴마사(광역) + 적 4종 추가 → 2~6장 스테이지 → 보스 패턴 심화 → 엔딩 분기. 전부 **데이터 추가**가 주(主), 코드 추가는 종(從)이어야 한다(D4가 성공했다는 신호).

> 슬라이스 이후 추가될 동료(직업) 후보 풀은 [12] 동료 직업 로스터 참조(착호갑사·도사·승병·추노객·포수·의원·광대·장례사 등, 전부 후속/예약). 고유 능동 기믹이 현 엔진 모델 밖인 항목([12]§4 예약 표)은 데이터 추가만으로 안 되며 개별 결정이 선행돼야 한다.
